# Building a chrischall fleet MCP

The fleet is ~19 sibling MCP servers under `~/git/*-mcp`, all on the same skeleton and sharing `@chrischall/mcp-utils` (generic scaffolding), `@fetchproxy/server` (browser-bridge HTTP for sites without an API), and `@chrischall/realty-core` (realty math). Build new ones the same way; don't reinvent the glue.

Canonical examples to copy from: **splitwise-mcp** (bearer/direct-API archetype), **artsonia-mcp** (cookie-session / username+password archetype), **redfin-mcp** / **zillow-mcp** (fetchproxy archetype), **musicbrainz-mcp** (rate-limited public API — no-auth reads + OAuth-grant writes).

## Deliverable: fpx skill first, full MCP only on request

**Default to a lightweight shell-out skill, not a full MCP — always produce it, even when a full MCP is also requested. Its usual tool is `fpx`; use `curl`/node-fetch when the service is reachable server-side (see below).** `@fetchproxy/cli` (binary `fpx`) wraps the same browser bridge the fetchproxy archetype uses as a standalone one-shot CLI: point it at a URL or GraphQL endpoint and it fetches through the user's signed-in tab, no MCP process to run. For most "give me access to `<site>`'s data" asks, a shell-out skill is the whole job. **Match the tool to reachability:** use `fpx` only for the fetchproxy cases — bot-walled sites, or where the only credential is the live browser session — because every `fpx` verb routes through the signed-in tab. For anything reachable server-side (a public or bearer/token API), the skill shells out to plain `curl`/node-fetch with the key in a header — no bridge, no extension, no signed-in tab; routing a public endpoint through the tab buys nothing. Same lightweight skill tier either way; `fpx` is just the tool for the walled ones. (tripadvisor's Terra API = a `curl` skill; its no-key public-page fallback = the `fpx` bridge.)

Decide this BEFORE touching the skeleton or picking an archetype:

- **Not dictated** (user said "add `<site>`" / "I want `<site>` data" / "integrate `<site>`"): build the fpx skill, then **ASK** whether they also want a full MCP — say it's a much larger build (typed tools, normalization, tests, packaging, registry/plugin/CI) and most asks don't need it. Don't start the MCP until they say yes.
- **Full MCP dictated** ("build me a `<site>`-mcp", "make the MCP server"): **CONFIRM** before the full build — "that's the full server (~a day of build + release wiring); the fpx skill alone may be enough — want both, or just the skill?" — and **still ship the fpx skill too**, so they get both. The fpx skill is never skipped.
- **fpx skill dictated / "just a skill" / "without the MCP"**: do only the skill.

A full MCP earns its keep only when at least one of these is true — otherwise stop at the skill: **reachable from claude.ai / a phone / any client that isn't Claude Code on the box with the CLI** (a shell-out skill runs only where its `fpx`/`curl` binary lives — this is the single biggest reason to build the server, and the one a skill fundamentally can't cover; to actually DELIVER that reach, layer on the **hosted connector** — see §*Hosted connector* — when the auth is serverless-safe); typed/validated tools an agent calls without shelling out; normalized records + a library surface (e.g. realty-meta consumers); confirm-gated writes; a first-class healthcheck; distribution through the MCP registry / plugin marketplace. If none apply, the skill is the finished deliverable.

### Authoring the fpx skill

`@fetchproxy/cli` — install `npm i -g @fetchproxy/cli`; the user also needs the **Transporter** extension (same one the fleet's bridge uses). Model: one **profile** per service (`fpx profile add <name> --domain <apex>`), its own identity `fpx-<name>`, paired once (`fpx pair -p <name>` → approve the 6-digit code in Transporter; the trust persists across invocations). Verbs: `get` / `post-json` / `request` (fetch), `cookies` / `local-storage` / `session-storage` / `indexeddb` / `session` (reads), plus `health` / `pair`. `fpx profile declare <name> --cookie k --capture-header name@host` widens scope and forces a re-pair. **Output contract: stdout = data only** (pipe to `jq`), stderr = pair codes / status; fetch-verb exit codes `2` bridge down, `3` bot wall, `4` upstream non-2xx.

Author it with the `skill-creator` skill: a lean `SKILL.md` (one-time profile-add + pair; the core call pattern; any resolve-first rule; the exit codes) plus a `references/` file of ready-to-run request bodies with `jq` recipes. **Verify every request shape live before baking it in** (same rule as coding an MCP endpoint — capture the real request, don't guess). Ship it where the service's skills live: `<repo>/skills/<name>-fpx/` in a fleet repo (auto-discovered via `plugin.json` `"skills": "./skills/"`, and already covered by the package `files`), or a standalone skill dir otherwise. **Canonical example: `hemnet-mcp/skills/hemnet-fpx`** — anonymous GraphQL through the bridge, its `references/graphql-queries.md` carrying the live-verified query bodies.

**Authoring gotchas (from the 2026-07-13 fleet-wide build of 25 access skills):**

- **Pick the tool from the MCP's `client.ts`, not its archetype.** A repo that *depends* on `@fetchproxy/server` may bridge only ONE call (the auth/token mint) and curl everything else — resy's `api.resy.com` isn't bot-walled, so its skill is curl + a one-time `fpx post-json` token bootstrap, not full fpx. Read the client before committing to fpx vs curl vs hybrid.
- **When an MCP already exists, derive from it — don't re-capture.** Its `src/client.ts` / `src/tools/*.ts` / `docs/<SVC>-API.md` ARE the live-verified shapes; transcribe them (the "capture live" rule is for net-new services with no MCP). For HTML-parse-heavy sites, reuse the MCP's own parser via `node --input-type=module -e "import {parseX} from '<repo>/dist/parse.js'; …" | jq` rather than re-deriving brittle CSS selectors (artsonia).
- **SSR JS-object stores need a ported extractor, not `jq`.** Sites that embed an entity-encoded / bare-`undefined` JS-object store (`__NEXT_DATA__`, `$REDUX_STATE`, `window.__store__`, a global assign) can't be read by `jq`. Port the MCP's brace-match + entity-decode into a small dependency-free `.mjs` shipped in the skill's `references/`, verified against a synthetic fixture before baking the recipes (musescore/tock/compass/zillow).
- **Know fpx's boundaries — it is NOT the full bridge.** `fpx` does `get`/`post-json`/`request` + cookie/localStorage/sessionStorage/indexedDb/`session` reads + `health`/`pair`. It has **no DOM-read and no file-download verb**, so an fpx skill CANNOT complete a Cloudflare-Turnstile / DOM-token write flow (easytable) or fetch file bytes behind a cross-origin/attachment download (musescore scores, alltrails GPX) — it can only *resolve* such URLs. Document the limit; never fake it. (Candidate `@fetchproxy/cli` additions: a `dom` read verb, a `download` verb.)
- **Two packaging checks.** (1) `package.json` `files` must include `"skills"` — many repos list only `dist`/`SKILL.md`, so a new skill silently wouldn't ship on npm. (2) `.claude-plugin/plugin.json` `"skills"`: canonical is `"./skills/"` (hemnet); if it points at `"./SKILL.md"` **and a root `./SKILL.md` exists**, LEAVE it (the skill still ships on npm but isn't plugin-auto-discovered until a separate root-skill migration) — repoint to `"./skills/"` only when the pointer is broken (no root skill to displace). Never move a root `SKILL.md` as a side effect.

## Skeleton

```
src/
  index.ts        # bootstrap — runMcp({ name, version, banner, deps, tools })
  client.ts       # API client (reads creds, talks to the service)
  tools/<area>.ts # registerXxxTools(server, deps) → server.registerTool(...)
tests/            # vitest; no real network (mock fetch / the bridge)
  version-sync.test.ts   # versionSyncTest from @chrischall/mcp-utils/test
SKILL.md, manifest.json, server.json, .mcp.json, .claude-plugin/  # packaging
.github/workflows/   # ci, release-please / tag-and-bump, pr-auto-review, auto-merge
```

Each `tools/*.ts` exports `registerXxxTools(server, deps)` that calls `server.registerTool(name, { description, annotations, inputSchema }, handler)` (high-level `McpServer` API with zod). `index.ts` only wires them.

## Bootstrap (index.ts) — both archetypes

```ts
#!/usr/bin/env node
import { runMcp } from '@chrischall/mcp-utils';
import { FooClient } from './client.js';
import { registerThingTools } from './tools/thing.js';

// Build the client in the CALLER so the deferred-config-error pattern holds:
// the server still boots (and answers the host's install-time tools/list probe)
// when creds are absent — the error surfaces on the first tool call instead.
const client = new FooClient();

await runMcp({
  name: 'foo-mcp',
  version: '1.0.0', // x-release-please-version
  banner: '[foo-mcp] This project was developed and is maintained by AI. Use at your own discretion.',
  deps: client,
  tools: [registerThingTools],
});
```

**Deferred-config-error pattern** (do this in `client.ts`): the constructor reads creds via `readEnvVar`; if missing, store a `configError` instead of throwing, and re-throw it from a `requireKey()` called at request time. Lets the server start without creds. **It can be PARTIAL:** if reads need no creds but writes do (the rate-limited-public-API archetype), defer only the write-side `configError` — reads always boot, the error surfaces on the first write.

## Archetypes

**Auth preference order — avoid requiring the browser bridge at runtime.** fetchproxy needs the Transporter extension + a signed-in tab, so it's the heaviest thing to ask of a user. Pick the FIRST that authenticates cleanly, and even then keep the bridge off the hot path:

1. **username/password in `.env`** — a real server-side login (the cookie-session archetype below; or a bearer/token if the API offers one). No browser, no extension. Always try this first.
2. **fetchproxy *bootstrap*** — use the bridge ONCE to capture auth from the signed-in tab (its session cookie / storage, via `createBootstrapOpts({ bootstrap: { cookieKeys: [...] } })` → the `read_cookies`/`read_local_storage` capabilities), then make every actual request with plain node `fetch` carrying that cookie. The bridge touches only the handshake, not the workload. **Bot-walled + no server-side login variant** (DataDome/PerimeterX-fronted, NO username/password form and NO token exchange — the only credential is what the browser's own API requests carry): capture the **request headers verbatim** via `createBootstrapOpts({ bootstrap: { captureHeaders: [{ host, path: '/api/*', headerName: 'cookie' }, …any app-key header like `x-at-key`] } })` (the `capture_request_header` capability → `session.capturedHeaders`). Snapshotting the live `Cookie` header (incl. the short-lived anti-bot cookie) beats enumerating named `cookieKeys` you can't fully see, and grabs request-only headers (app keys) that aren't in any store. Drive it with **`CookieSessionManager`** whose `isExpired` treats **403 as well as 401** as expiry — a 403 = the anti-bot cookie went stale → re-capture — and cache the no-config error via `isPermanentError` so a missing session doesn't re-run the bridge on every call. A non-JSON 2xx read is almost always a bot-challenge interstitial: throw an actionable "refresh a signed-in tab" error, don't `JSON.parse` it blind. (ofw-mcp / alltrails-mcp.)
3. **full-on fetchproxy** — every request routed through the signed-in tab. Only when the site can't be reached server-side at all (a hard bot-wall on every endpoint).
4. **other** — if none of the above authenticate cleanly, propose changes (to `@fetchproxy/server` / the bridge, or a different capture) to get a working *minimal-bridge* auth; don't silently settle for full-on fetchproxy.

**Whichever you land on, use node `fetch`/GET for everything the bridge isn't strictly needed for** — route only the unavoidable calls (auth, or specific walled endpoints) through fetchproxy and node-fetch the rest. Don't reach for fetchproxy just because a sibling did; reach for it only when server-side auth/fetch genuinely fails.

- **bearer / direct API** (splitwise, tempo, ioffice, app-store-connect, skylight, gemini-mcp): `client.ts` does `fetch` with a bearer/token header. Use `createApiClient` if it fits; always route error bodies through `formatApiError`/`truncateErrorMessage`. No fetchproxy. **Non-`Authorization: Bearer` auth → write a thin custom `fetch` client instead of `createApiClient`** — its `getToken` only emits `Authorization: Bearer …`. APIs with a custom auth header (`x-goog-api-key`), a model/id baked into the path (`/models/{m}:generateContent`), or binary-in-JSON responses are cleaner as a small `call()` that still uses `McpToolError` + `truncateErrorMessage` (gemini-mcp). **Binary-OUTPUT tools** (image/file generation): write bytes to a configurable dir (`<SVC>_OUTPUT_DIR` → cwd) and return the path, with an `inline` flag for base64 via `imageResult`, and non-overwriting filenames — don't dump megabytes of base64 into the result by default. These create local files / cost money but don't mutate remote state, so they are **not** `confirm`-gated (confirm is for remote writes). **Tiered APIs can answer `401` (NOT `402`/`403`) for endpoints ABOVE the key's plan** — FlightAware AeroAPI returns `401` titled "Invalid API key" with a *tier* detail for alerts/history/canonical on the free tier, so a VALID key looks "invalid." `createApiClient`'s `onUnauthorized`/`onRateLimited` take **NO args** (they can't see the response body), so don't let `onUnauthorized` flatly assert "key invalid" — word it to name BOTH causes (bad key OR insufficient tier) and mark the tier-gated tools' descriptions. **Billed / rate-limited reads → a small per-path response cache** (in-memory, TTL via `<SVC>_CACHE_TTL`, bounded + oldest-evicted, writes never cached) cuts duplicate spend; give *reference* data (airport/operator metadata, canonical lookups, routes) a much longer **second tier** (`<SVC>_STATIC_CACHE_TTL`, e.g. 1h vs 15s) selected per-tool via `get(path, { cache: 'static' })` (flightaware-mcp).
- **cookie-session / username+password** (artsonia; many school/portal sites): a classic server-rendered `.asp`/`.php` site — no public API, no JSON store, no bot-wall — that accepts a real **username/password form POST** and hands back an **HttpOnly session cookie**. No browser bridge needed. `src/auth.ts` posts the login form and captures all `Set-Cookie`s into a cookie jar (use `parseCookieJar`); `src/client.ts` fetches with the jar over plain node `fetch` (`redirect: 'manual'` on the login POST so the 302's `Set-Cookie` is readable). Detect session expiry by a redirect back to the login page → **single-flight re-login** (one shared in-flight promise, à la `TokenManager`). Deferred-config reads `<SVC>_USERNAME`/`<SVC>_PASSWORD`. Parse the server-rendered HTML with `node-html-parser`. Keep `@fetchproxy/server` only as an OPTIONAL, env-selected fallback — not the default. See the **cookie-session / classic-form specifics** below.
- **fetchproxy / browser-bridge** (realty cohort, reservations, finance, school): the site has no public API AND can't be authed/fetched server-side (try options 1–2 above first), so requests go through a signed-in browser tab via `@fetchproxy/server`. `src/transport-fetchproxy.ts` wraps it; session tools expose/select accounts. Use `@chrischall/mcp-utils/fetchproxy` **only on `@fetchproxy/server` >= 0.11** (it re-exports 0.11+ APIs). Repos on older pins keep their own transport until a deliberate bump. Even here, node-fetch any endpoint that doesn't actually need the bridge.
- **rate-limited public API — no-auth reads + OAuth-grant writes** (musicbrainz): a real public REST API (no key needed for reads, just a descriptive **`User-Agent`**) with a **hard rate limit** and OAuth2 only for writes. `client.ts` is a thin custom client (not `createApiClient` — see the throttle/503 gotcha) with one `get()` (no auth) and one `write()` (bearer + `client=` param). **Partial deferred-config:** reads boot with NO creds, so there's no read-side `configError`; only the OAuth `write()` path defers its config error (missing `<SVC>_OAUTH_CLIENT_ID/SECRET/REFRESH_TOKEN`) to the first write. Confirm-gated writes route through `write()`. See the **rate-limited / OAuth-writes specifics** below.

### fetchproxy specifics (hard-won — each was a real failure in the musescore-mcp build)

- **One shared port: `37149`.** The whole fetchproxy fleet binds the SAME concentrator port — the **Transporter** browser extension dials that ONE port, and servers host/peer-elect on it (first to bind is host; the rest are peers tunneling through it). A new fetchproxy MCP MUST default to `37149` (`DEFAULT_PORT = 37_149`; env override `<SVC>_WS_PORT`). Give it a "unique" port and the extension never connects (`ws://127.0.0.1:NNNNN … ERR_CONNECTION_REFUSED`) — it has no idea your port exists. Copy the port from a sibling (zillow/redfin); don't pick a new one.
- **Lazy bind + one-time pairing.** `listen()` reserves nothing; the port binds + role-elects on the FIRST verb call (or `connect()`). A brand-new server's identity (`~/.fetchproxy/identity/<server>.json`, auto-created) is unknown to the extension, so the first request returns `pairing required … pair code: NNN-NNN` — the user approves it ONCE in the Transporter popup (TOFU, per-identity, persists). Onboarding a new fetchproxy MCP = (1) default port 37149, (2) trigger one call (`<svc>_healthcheck`), (3) approve the pair code in Transporter.
- **Parse the SSR JSON store; verify against RAW-FETCHED bytes, not the hydrated DOM.** Modern SPAs render NO cards/tables in the fetched HTML (client-hydrated) — the data is an entity-encoded JSON store (a `__NEXT_DATA__` analogue, often attribute-escaped: `&quot;`/`&amp;`). The MCP parses what fetchproxy fetches = raw SSR, which ≠ `document.querySelectorAll` in a live tab. Verify your parser against the actual bytes: in a signed-in tab run `await fetch(path,{credentials:'include'}).then(r=>r.text())` and parse THAT. Recover-from-entities + bracket-match (don't regex nested JSON); skip any bot-manager fields (`bm_*`, `client_ip`, `ja3`/`ja4`) that ride along in the store.
- **Tighten Cloudflare detection.** The shared `classifyBotWall` covers perimeterx/datadome; for Cloudflare-walled sites detect the interstitial by DEFINITIVE markers only — `_cf_chl_opt` and `<title>Just a moment`. Do NOT match `cdn-cgi/challenge-platform` / `challenges.cloudflare.com` (Cloudflare inlines those on CLEARED pages too) or gate on body size — that false-positives on legit pages under ~80 KB (it made every score-page fetch wrongly report a bot-wall while the bigger search pages sailed through, so "search works but detail pages fail" is the tell).
- **The bridge does `fetch()` — it can't navigate, follow cross-origin redirects, or stream binaries.** Endpoints that only work as a browser NAVIGATION — file downloads with `Content-Disposition: attachment`, or a same-origin endpoint that 302s cross-origin to a presigned S3 URL — can't be fetched: the bridge's `fetch` throws on the cross-origin redirect, and a server-side node `fetch` hits the Cloudflare wall (403). Such a tool can only **resolve** the URL (return it for the user to open), not download the file. Don't scrape the page's render tiles to fake it.
- **Capture response SHAPES through the bridge itself, not by replaying the captured cookie from Node.** On a cookie-bootstrap MCP (alltrails), DataDome can `403` EVERY Node replay of a freshly captured cookie on a day when an in-tab fetch sails through — a shape-capture probe built on the real client just loops capture→replay→403. Probe via `createFetchproxyTransport({serverName, domains, defaultSubdomain})` → `requestJson(method, path, {headers})` instead: the request runs as a same-origin in-browser fetch inside the signed-in tab, so the bot wall never sees Node. Two details: send the site's app-key headers (`x-at-key` etc.) explicitly — an in-tab `fetch` does NOT add them; and when a bootstrap capture window sits at "waiting for next request", `open "https://<site>/…"` from Bash feeds it (the page load fires the API calls). The alltrails captures done this way immediately exposed a live bug (`/me` returns `{users:[…]}`, not the `{user:…}` the code read) — which is the point of capturing over guessing.
- **Verify a real SUCCESS, at the right layer, before coding the gate.** Downloads/navigations DON'T appear in XHR/`fetch`/HAR monitoring — watching those and concluding "nothing happens" is wrong. Confirm an actual artifact end-to-end (a file lands on disk) and read the JSON store for the real gate field before encoding it. (Cost of skipping this: a download gate built on the wrong field — `hasAccess`, an entitlement flag — instead of `is_free`, which refused every actually-downloadable score.)

### cookie-session / classic-form specifics (hard-won — the artsonia-mcp build)

- **Capture a JS-driven login by instrumenting it, not from static HTML.** Modern login forms inject the `Username`/`Password` `<input>`s via JS, so the raw HTML has only hidden fields. You can't type a password yourself (safety rule) — capture the real request by hooking `submit`/`fetch`/XHR while the USER logs in once, then replicate the form POST. (Artsonia: `POST /members/login.asp`, body `Username&Password&TargetUrl&Action=login`, `redirect:'manual'` to read the 302's HttpOnly cookie; no CSRF on these old stacks.)
- **A 302 / redirect-away is NOT proof a write persisted.** Old form handlers 302 to a content page on BOTH a real save and a silently-ignored one. Verify every write by RE-READING the resource — never trust the status code. (Cost of skipping: `set_notifications` reported success on a 302 that saved nothing.)
- **Submit a checkbox's real `value`, not `"on"`.** Classic checkboxes carry `value="Y"`; sending `name=on` is silently dropped. Capture each control's real value — checked → `name=<value>`, unchecked → omit it (as the browser does).
- **One field in a master form = read-modify-write.** If a toggle lives in a big profile form, re-POST the WHOLE form: re-send every current field verbatim (hidden ones at their real values, e.g. `DidChangePassword=N` not `0`), BLANK the password fields, flip only your target. Never send a password.
- **Verify INNER selectors against the live DOM, not just the container.** The data's in the server-rendered HTML (parse with `node-html-parser`) — top-level selectors (`.artist-card`) confirm trivially, but the child selectors you guess for a fixture (`.name`, `.date`) are usually wrong. Run the real parser logic against `await fetch(path,{credentials:'include'}).then(r=>r.text())` and diff the output before trusting it.
- **Recon through claude-in-chrome redacts secrets.** It blocks content that looks like cookies/query-strings/tokens (`[BLOCKED: …]`), so extract STRUCTURE — tag/class trees, form field NAMES, URL path + param KEYS — never raw HTML/values, and never log a captured password/cookie. Media is often a public CDN (fetchable server-side, no auth) whose `Last-Modified` header is a free per-item date when the HTML exposes none.

### rate-limited public-API / OAuth-writes specifics (hard-won — the musicbrainz-mcp build)

- **A hard rate limit needs a PROACTIVE throttle, not just retry — and `createApiClient` only retries 429.** APIs that cap you (e.g. MusicBrainz: 1 req/s) signal an overage with **503 + `Retry-After`**, NOT 429, so `createApiClient`'s built-in 429 retry never fires and it surfaces a raw error. Write a thin custom client whose every call (reads AND writes) funnels through one **serialized min-interval queue** (`src/throttle.ts`: a promise chain that spaces request *starts* ≥ limit apart, with injectable `now`/`sleep` for tests) so concurrent tool calls line up instead of bursting; also retry 503/429 honoring `Retry-After`. One throttle, one client — never hit the API outside it or you bypass the spacer. A descriptive `User-Agent` is mandatory (these APIs block generic/empty UAs).
- **OAuth-grant writes: `createOAuth2Refresher` + a tiny in-memory token cache.** For the write path, `createOAuth2Refresher({ endpoint, refreshToken, params:{client_id,client_secret} })` does the `refresh_token` grant; cache the `{accessToken, expiresAt}` and refresh ~60s before expiry (the refresher already shares a single in-flight exchange, so concurrent writes don't double-refresh). The **human must register the app and supply `client_id`/`secret`/`refresh_token`** (an agent never sets creds), and **live write-verification is GATED on that** — unit-test the write shapes meanwhile.
- **Capturing the refresh token: verify redirect rules against the LIVE registration form, then use a `http://localhost` loopback listener.** The provider's OAuth *doc* is often stale (MusicBrainz's said `oob`/`localhost`; the live form rejected both, demanding a **reverse-DNS** scheme for "installed apps") — read the actual form, not the doc. You can't type the user's password or click Allow for them, and a custom-scheme (`app://…`) redirect can't be captured (Safari won't expose the redirected URL). So: generate the authorize URL **with PKCE** (`code_challenge=S256`), run a one-shot `http://localhost:<port>/` listener that captures `?code=`, exchanges it (`grant_type=authorization_code` + `code_verifier`), and writes the refresh token into `.env`. The human just clicks Allow.
- **Write responses may be NON-JSON** — MusicBrainz answers a successful write with XML / `{"message":"OK"}`, so `write()` returns the **raw body**; don't blindly `JSON.parse` it (keep the same `formatApiError`/`McpToolError` handling for non-2xx).
- **The re-read used to verify a write can itself be CACHED.** Re-reading after a write (the skill's "a 200 is not proof, re-read" rule) can serve a **stale cached body** — a `/ws/2` GET showed the old state right after a successful write/withdraw, so the write looked like it failed when it hadn't. **Bust the cache with a unique nonce query param** (`&_=<nonce>`) on the verification read. (Read tools are unaffected — this only bites write-verification re-reads.)

## Hosted connector (Cloudflare Worker) — optional dual-target, on request

A **hosted remote connector** makes the same MCP reachable from **claude.ai (web/desktop/mobile)** — the "reachable from a client that isn't Claude Code on the box" reason to build a server, now actually delivered. It's a SECOND entry point (`src/worker.ts`) that wraps the SAME tool registrars in a Cloudflare Worker with OAuth login + streamable-HTTP/SSE, via the shared **`@chrischall/mcp-connector`** npm harness. The local stdio server stays unchanged. **Build it only when the user asks for hosted / claude.ai access** — it's a deploy target, not a default.

**Appropriate ONLY when the auth works from a serverless runtime — NO browser bridge, NO ephemeral session cookie.** The Worker has no Transporter extension, no signed-in tab, no filesystem, and `import.meta.url` / Node globals behave differently. So:
- ✅ **bearer / API-key** archetype (splitwise, setlist, untappd-xauth) — the login page collects a long-lived key; the simplest connector.
- ✅ **username/password with a real server-side login** (ofw) — the login page collects creds. If the minted token has a short TTL and NO refresh token, store the creds themselves in the OAuth props (encrypted at rest in `OAUTH_KV`) so the per-user client can silently re-login, and say so HONESTLY in the `privacyNote` (don't claim "not stored" when you store them).
- ❌ **fetchproxy / browser-bridge** archetypes — impossible: the bridge needs the extension + a signed-in tab that don't exist server-side. Don't build a connector for these.
- ❌ **ephemeral session-cookie writes with no server-side re-login** — the cookie goes stale mid-session with no hosted refresh path. **Mixed repo → ship a reduced connector:** register only the serverless-safe tools (the API-key / read subset), OMIT the cookie/fetchproxy tools from the Worker's `tools` array, and keep that code out of the Worker bundle (it stays behind the stdio lazy-import boundary). setlist shipped **read-only v1** this way — its 2 attendance-write tools (cookie session) are excluded.

**One Worker per MCP — do NOT build a consolidated "hub".** (Researched: `@cloudflare/workers-oauth-provider` hardcodes a single `OAUTH_KV` binding, so a host-routed hub would share ONE token keyspace across services — *weaker* isolation, not stronger — plus shared deploy blast radius + deploy coupling. Both topologies are equally capable on Cloudflare; per-Worker wins on coupling.) Each connector = its own Worker, own hostname `connector.<svc>.nullnet.app`, own `OAUTH_KV` namespace, own DO namespace, own deploy. De-duplicate the harness by **publishing `@chrischall/mcp-connector` to npm** (its own repo, peer-deps `agents`/`@modelcontextprotocol/sdk`/`@cloudflare/workers-oauth-provider`), NOT by vendoring it into each repo.

**Canonical examples: ofw-mcp** (stateful — password re-login + a Durable-Object message cache), **untappd-mcp** (xauth + cache DO), **splitwise-mcp** (API-key, stateless, full-write), **setlist-mcp** (API-key, stateless, read-only v1). Copy a stateless one for anything without a local cache.

### Build shape (mirror ofw-mcp)

1. **Prereq — transport-neutral registrars + injectable client (land as its own mergeable PR first).** The skeleton already passes `(server, deps)`, so most repos are close. But if a tool file closes over a MODULE-LEVEL singleton client internally, refactor: every `registerXTools(server, client)` takes the client as an ARG (import only its TYPE), and the client gets an OPTIONAL constructor seam (`new Client({ apiKey })`) so a per-user instance can be built — KEEP the env-driven stdio singleton export unchanged. Pure cleanup; unblocks the Worker.
2. **`src/<svc>-auth.ts`** — a `ConnectorAuth<Props>` (`import type { ConnectorAuth } from '@chrischall/mcp-connector'` — type-only so the file stays node-loadable): `{ service, accent, privacyNote, fields:[{name,label,type?}], login(fields, env): Promise<Props> }`. `login()` VERIFIES the creds (build the client, hit a cheap current-user / read endpoint, throw on failure → surfaced on the login page) and returns the props to persist.
3. **`src/worker.ts`** — `const { Agent, handler } = createConnector<Props, Client>({ name, version, auth, buildClient: (props, env) => new Client({ ...props }), tools: [...registrars] }); export { Agent as <Svc>McpAgent }; export default handler;`. Register the tools appropriate to the connector's scope (all, or a read-only subset).
4. **`wrangler.jsonc`** — `name: "<svc>-connector"`, `main: src/worker.ts`, `compatibility_flags: ["nodejs_compat"]`, a `MCP_OBJECT` Durable Object → `<Svc>McpAgent` with a `v1` `new_sqlite_classes` migration, an `OAUTH_KV` namespace (placeholder id), a custom-domain `routes` entry `connector.<svc>.nullnet.app`, and any `vars` the client's env-gates read (`nodejs_compat` populates `process.env` from `vars`, so existing write-mode / config env helpers work unchanged in the Worker). **Stateless service → NO cache DO.** Stateful (a local SQLite cache) → split the cache into a driver-agnostic core + a `node:sqlite` backend + a Durable-Object (`ctx.storage.sql`) backend, add a `CACHE_DO` keyed by the authenticated username, and inject the store into the registrars (ofw's `src/cache/{store,node,durable}.ts` + a per-client WeakMap in `worker.ts`).
5. **Two vitest configs.** Node pool: `test.exclude` the worker test files + `coverage.exclude` `src/worker.ts` (and `src/cache/durable.ts` if present) — they import `cloudflare:workers`/`agents` and can't load under Node; NO node test may import them. Workers pool: `vitest.workers.config.ts` (`@cloudflare/vitest-pool-workers`, pointed at `wrangler.jsonc`) runs `tests/worker*.test.ts` (assert OAuth discovery, 401 on unauth `/mcp`, the `/authorize` login page, the registered tool roster; for a stateful connector, a DO-cache round-trip). **`tsconfig` MUST `exclude` `src/worker.ts`** so the stdio `tsc` build doesn't emit `dist/worker.js`.
6. **package.json** — `@chrischall/mcp-connector` `^x` (npm, NOT vendored) + its peer deps as devDeps (`agents`, `@cloudflare/workers-oauth-provider`, `@modelcontextprotocol/sdk`) + `wrangler` / `@cloudflare/vitest-pool-workers` / `@cloudflare/workers-types` + `worker:dev`/`worker:deploy`/`worker:test` scripts. Add `.dev.vars.example` + `docs/DEPLOY-CONNECTOR.md` (operator runbook). `server.json` does NOT advertise a `remotes` entry — the connector is a personal manual deploy, not a registry remote (mirror untappd).

### Hosted-connector gotchas (hard-won — the ofw → splitwise → setlist builds)

- **The `.env` module-init guard — bit EVERY connector.** `client.ts` loads `.env` at module top-level via `fileURLToPath(import.meta.url)`. In the DEPLOYED Worker `import.meta.url` is undefined → `fileURLToPath(undefined)` throws at Worker STARTUP VALIDATION (`wrangler deploy` fails `code: 10021`, "path argument must be … Received undefined"). **Neither usual gate catches it:** Miniflare (`worker:test`) provides `import.meta.url`, and `wrangler deploy --dry-run` only bundles (no startup validation) — ONLY a real `wrangler deploy` surfaces it. Wrap the load in `try/catch` (Node loads `.env` as before; the Worker skips it) as PART of the worker wiring — it's fleet-wide in every `client.ts`.
- **Disallowed global-scope ops in a module-singleton client constructor — the SECOND startup-validation trap, equally gate-invisible.** The fleet exports `export const client = new Client()` at module scope; when `worker.ts`'s graph loads `client.ts`, that singleton is constructed IN GLOBAL SCOPE, and the Workers runtime FORBIDS async I/O (`fetch`/`connect`), `setTimeout`, and **random-value generation** there (`wrangler deploy` fails `code: 10021`, "Disallowed operation called within global scope … generating random values"). ofw/splitwise/setlist slipped by only because their constructors were side-effect-free; **zola's called `crypto.randomUUID()`** (a `deviceSessionId` field) → crash. Same false-green as the `.env` trap (`worker:test` + `--dry-run` both pass; only a real deploy validates global scope). **Rule: a module-singleton client's constructor must be PURE** — lazy-init anything touching I/O / random / timers via a memoized getter (`private get deviceSessionId() { return (this._id ??= crypto.randomUUID().toUpperCase()); }`). (zola.)
- **A DETACHED `globalThis.fetch` breaks every request in the Worker — the THIRD gate-invisible trap, and the only one that survives a successful deploy.** A client that stores the global fetch as a property and calls it detached (`this.fetchImpl = opts.fetchImpl ?? globalThis.fetch` → `this.fetchImpl(url)`) throws **`Illegal invocation: function called with incorrect \`this\` reference`** in workerd, which requires `fetch` to be invoked with `this === globalThis`. Node has no such rule, so EVERY node test passes; `wrangler deploy` succeeds (bundling never invokes fetch); and a `worker:test` that only asserts OAuth discovery / the login page never makes an outbound request either. It surfaces only when a real fetch runs — for charlotteonthecheap the login page reported the site as *unreachable* while it was plainly up, and no tool could have fetched anything. **The exact form matters** (verified 2026-07-19 against the deployed setlist connector): `?? fetch` — the BARE global identifier, which is what mcp-utils' `createApiClient` uses (`const doFetch = opts.fetchImpl ?? fetch`) — is **safe**, so the fleet's `createApiClient` consumers are unaffected; only the `globalThis.fetch` property form detaches. Fix by calling it as a method: `opts.fetchImpl ?? ((input, init) => globalThis.fetch(input, init))` (also keeps a `vi.spyOn(globalThis,'fetch')` working). **Guard it in the WORKERS pool** with a test that exercises a real client request — the binding error throws before any network I/O, so asserting the failure reason is never `/illegal invocation/i` is robust even with no CI egress. Diagnose this class of bug with a differential `wrangler dev` run: still failing from a residential IP means a code/workerd issue, not an upstream IP block — and don't let a client's catch-all swallow the underlying error, or a `this`-binding bug masquerades as a site outage.
- **Deploy is MANUAL, in the operator's Cloudflare account — no CI deploy** (matches untappd/ofw). Needs a token with **Workers Scripts:Edit + Workers KV Storage:Edit** (the "Edit Cloudflare Workers" template) or `wrangler login`; a read-only / zone-only token fails KV-create + deploy with `code: 10000` auth errors. Runbook: authenticate → `wrangler kv namespace create <svc>-connector-oauth` → paste the id into `wrangler.jsonc`'s `OAUTH_KV` → `npm run worker:deploy`.
- **Each connector needs its OWN `OAUTH_KV` namespace** — give it a distinct TITLE (`<svc>-connector-oauth`); the binding NAME stays `OAUTH_KV`, only the id differs. Two connectors sharing one namespace cross-wire their OAuth grants. (`create OAUTH_KV` with the bare title collides across the account.)
- **Custom domain**: the zone (`nullnet.app`) must be in the deploying account; the edge TLS cert provisions a few minutes AFTER deploy (`https` → connection-refused / TLS handshake fail meanwhile — normal). Use the `*.workers.dev` URL immediately; the custom domain self-heals. Verify with the `/register` → `/authorize` flow (a bogus `client_id` 500s — expected; claude.ai does DCR first).
- **`agents`' optional `ai@^6` peer** ERESOLVEs against a repo that depends on `ai@^7` (untappd) — add a repo-root `.npmrc` `legacy-peer-deps=true` (benign optional-peer skew; we don't use agents' `ai` path). Repos with no `ai` dep install cleanly.
- **No filesystem on the Worker.** File-writing / -reading tools (download-to-disk, upload-from-a-path) must go inline-only (bytes as MCP content) or throw a helpful "not available on the hosted connector" error — abstract the FS behind an injectable boundary so stdio keeps disk I/O (ofw's `AttachmentIO`; artsonia's `DownloadIO` → disk-backed `NodeDownloadIO` for stdio vs `InlineDownloadIO` returning base64 image blocks on the Worker). Two honesty/size traps ride along: (a) the inline IO can only surface SOME outputs (image bytes) — sidecars/manifests (`.json`, `index.json`) have nowhere to go, so a tool that reports `wrote <path>` / counts must OMIT those fields on the hosted path (give the IO a `persistsFiles` capability flag and gate the write + the summary field on it), not advertise files it never wrote; and (b) an unbounded inline pull (every full-res image base64'd, +33%, in one response) can blow past a host's response-size limit — cap cumulative inline bytes and surface a text note for what was dropped, don't silently truncate.
- **The injected inline IO is a per-session SINGLETON — reset any accumulated state per call, or it leaks across invocations.** `McpAgent` registers tools ONCE per `<Svc>McpAgent` Durable Object, and the DO persists for the whole MCP session, so the single IO instance you construct in `worker.ts` (`new InlineDownloadIO()`) is REUSED by every tool call. Any buffer it accumulates (e.g. the image content blocks it returns) must DRAIN on read — the 2nd call otherwise returns the 1st call's bytes plus its own, the 3rd all three, etc. **False-green, like the other startup traps:** the stdio IO is stateless (`NodeDownloadIO` writes straight to disk, returns `[]`), so the whole node suite passes; only a repeated hosted call reveals it. Cover it with a two-invocation test asserting the second result carries ONLY its own output. (artsonia's `InlineDownloadIO.extraContent()` drains its buffer.)
- **Auto-merge orphans your follow-up commit — hits connector work HARD** because it's a chain of small PRs (refactor → worker → env-guard fix → KV-id) landing minutes apart. The KV-id and the `.env` guard routinely need to land AFTER the worker PR already squash-merged, so they go on FRESH branches off the new `main` (see the two auto-merge gotchas in §Gotchas). Expect it; never push a follow-up to an already-merged branch.

## Conventions (how chris likes them)

- **TDD, always.** Failing test → minimal code → green. Especially for write tools.
- **Verify endpoints before building them — at the right layer.** For no-API sites, capture the real request (DevTools → Network → "Copy as cURL", or a HAR) and pin the request *shape* in `docs/<SERVICE>-API.md`. Never code a write against an assumed body, and never encode a gate field you haven't seen succeed: confirm an actual successful artifact end-to-end (a real file/row), because downloads and navigations don't show up in XHR/`fetch`/HAR. Extract only the shape — **never commit captured cookies/tokens/signatures** (secret-scan before committing). **Premium APIs need a funded account to verify** — image/LLM endpoints (gemini-mcp's `gemini-3-pro-image`) return `429 … free_tier_requests, limit: 0` on a free key, so the verification gate is BLOCKED until the human enables billing; surface that and wait rather than guessing the shape. Also pin the SUCCESS response from a real 200 — request and response casing can differ (Gemini accepts snake_case `inline_data` on the request but returns camelCase `inlineData`), and a model/endpoint may only exist on `v1beta`, not `v1`. **Re-verify for EVERY new knob and especially every new/Beta endpoint, not just the first build** — a doc scrape (and a WebFetch summarized by a small model) is frequently wrong or incomplete: confirm each field name, accepted enum values, required headers (the Gemini Interactions API needs `Api-Revision: 2026-05-20`), and output constraints (it only accepts `image/jpeg`, not png) with a real call before coding. Cheap knobs (a new `imageSize` value, `thinkingConfig`, `seed`) are one curl each; a whole new endpoint earns a full request+response capture pinned in `docs/<SERVICE>-API.md`. **Sibling endpoints can use DIFFERENT input grammars — don't assume they share one:** AeroAPI's `/flights/search` + `/search/count` take a simplified `-key value` string, while `/search/advanced` + `/search/positions` take a structured `{op key value}` one (and the field names differ — `orig`/`dest`, not `origin`); mixing them `400`s. **Live-verify through the BUILT client, not just raw curl** — a throwaway read-only probe that `import`s `dist/client.js` and calls each tool's path exercises the real path-building + cache + error layers end-to-end (it caught the wrong-grammar example a mocked unit test never would), and it confirms tier-gated endpoints `401` gracefully. **Space the probe calls** (~3–4s apart): a billed/rate-limited API will `429` a burst of a dozen and mask the real results.
- **A documented OpenAPI/Swagger API on a site is often a SELLER/partner surface you can't actually use — find the CONSUMER surface before committing to an archetype.** Many portals (ticketing, real-estate, school) publish a clean REST API (e.g. `api.<svc>.com/v3` with a Swagger UI → `openapi.json`) that turns out to require **OAuth2 client-credentials / password-grant with venue/partner credentials a consumer doesn't have** — every endpoint 401s, even the ones labelled `/public/*`. The data a consumer can actually reach lives on the **www site's own internal API** (`/api/online/...`), which is typically **bot-walled** (DataDome/PerimeterX → fetchproxy) AND may return **encrypted/opaque payloads on its richest endpoints** (etix's full-text `/search` came back as base64 blobs; only the lighter `search/suggest` was clean JSON, and detail pages were SSR HTML w/ schema.org JSON-LD + microdata). So before picking bearer-vs-fetchproxy: probe BOTH the documented API (is it consumer-authable, or seller-only?) and the consumer site (reachable server-side, or walled? clean JSON, or encrypted/SSR?), and when the two diverge, **ask the user which surface they want** — a seller box-office MCP and a consumer-discovery MCP share almost no code (etix: confirmed seller-`v3`-OAuth vs. consumer-fetchproxy was a one-question fork that decided the whole build).
- **Confirm-gated writes.** Every mutating tool takes `confirm` (`schemaConfirm` from `zod`). Without `confirm: true` it makes **no** network call and returns a dry-run `preview()` of exactly what would be sent. Route every mutation through one private `client.write()` that attaches auth (cookie/CSRF or bearer) centrally.
- **Stream file uploads.** Never `readFileSync` a file just to wrap it in `new Blob([buf])` — use `fileBlob` / `readFileHead` from `@chrischall/mcp-utils` (or `fs.openAsBlob` directly) so a 20 MB upload isn't a 20 MB heap buffer.
- **Results:** `textResult(data)` for everything; errors via the typed `McpToolError` subclasses with an actionable `hint`.
- **Compact projection + drift-fallback for verbose reverse-engineered listings.** A reverse-engineered listing/detail endpoint often returns ~100 fat records — a lot of context for an agent that just wants to browse or rank. Add an opt-in `compact` arg that projects each item to a slim summary keyed off **documented** fields only, deriving convenience fields locally (e.g. metric→imperial) so the agent needn't convert. Validate the envelope with a **loose** zod schema that covers only the fields you read; when the expected array/shape isn't present, **warn to stderr and return the RAW response** rather than an empty/wrong projection — undocumented APIs drift, so degrade, never break. Keep `compact` default `false` so full records stay reachable. (alltrails-mcp.)
- **Secrets:** `.env` gitignored; throwaway-test writes go only to blackholed addresses (`@example.com`); only the user's own account/data.
- **Never merge PRs or add `ready-to-merge` yourself.** Squash-merge is the default; `pr-auto-review` + `auto-merge` ship it. On a `warn`/`fail` verdict, surface the findings and ask — don't override.
- **Describe what it does, neutrally.** Public descriptions (`marketplace.json` `metadata.description`, `server.json`, the GitHub repo About) state the *mechanism* — e.g. "routes through the user's signed-in `<site>` tab via the fetchproxy bridge, reusing their authenticated session" — not bot-evasion framing ("to dodge bot detection"). Same factual tone as CLAUDE.md's "every request rides the user's own browser session."

## Repo bootstrap — git, labels, release-please

A new fleet repo isn't done until ALL of this exists. Each line below was a real failure when missing (CI red, review step erroring, `main` unprotected, bundle bloat).

**Workflows** (`.github/workflows/`): the fleet uses **thin stubs that call the shared reusable workflows in [`chrischall/workflows`](https://github.com/chrischall/workflows)** — `ci.yml` → `reusable-mcp-ci.yml@main`, `pr-auto-review.yml` → `reusable-pr-auto-review.yml@main`, `auto-merge.yml` → `reusable-auto-merge.yml@main`; `release-please.yml` runs `release-please-action` then the `chrischall/workflows/.github/actions/mcp-publish@main` composite (npm `--provenance` + `.mcpb` + MCP registry + ClawHub — it **derives the package/skill name from the repo**, `package-name` defaults to the repo name, so NO hardcoded `skill_name`/`.mcpb` filename/`install-mcp-publisher` pin); `claude.yml` (`@claude` dispatch) stays inline; plus `dependabot.yml`. **Node:** `node-version: "26"`. **COPY THE STUBS FROM A FRESHLY-UPDATED SIBLING'S `origin/main`, NOT a local working tree** (`gh api repos/chrischall/<sibling>/contents/.github/workflows/<f>.yml --jq .content | base64 -d`) — a local clone can be STALE and hand you the OLD self-contained workflows (still work, but drift from the fleet, and dependabot then opens bumps for pinned actions the stubs no longer reference). The stubs are repo-agnostic — drop them in verbatim (node 26, `npm run build`/`npm test`, empty `conventions_hint`). The stub's job appears as `ci / ci`, but the required check is the `ci-gated` status either way (see the ruleset note below). **Register the new repo in `chrischall/workflows/fleet.json`** (just `{"repo": "chrischall/<name>"}` — defaults apply; override only `pat_secret`/`ci`/`node_version`/`test_command` per repo) so fleet-wide rollouts (`scripts/rollout.sh`, stub-trigger sweeps) include it — a brand-new repo is easy to forget there, and then it silently misses every later fleet change.

**Labels** — per-repo (`chrischall` is a *User* account: no org-wide labels/secrets). Create `auto-review`, `ready-to-merge`, `review-with-opus`, `release-ready` (the shared `reusable-pr-auto-review` gates the release-please PR's review on this — missing it just means the release PR never auto-reviews), `autorelease: pending`, `autorelease: tagged`, plus dependabot categories (`ci`, `security`, `test`, `javascript`, `github_actions`, `ignore-for-release`). Missing `auto-review`/`ready-to-merge` → pr-auto-review's first step (`gh pr edit --add-label auto-review`) errors and arming can't happen.

**Branch protection** — two rulesets on `~DEFAULT_BRANCH` (personal account uses rulesets, not classic protection): (1) block `deletion` + `non_fast_forward`; (2) require a PR + the required check, which is the **`ci-gated` COMMIT STATUS — a status context, not a job name.** (Verified across the fleet 2026-07-19: every live ruleset requires `ci-gated`.) Getting this wrong is silent and total — a ruleset requiring a context that nothing posts blocks every PR forever — so copy `ci-gated` verbatim and never put a job name here.

**Two shapes of `ci.yml` both post `ci-gated`**, so don't infer the gate from the workflow's shape: (a) the thin stub calling `reusable-mcp-ci.yml` with `gate-mode: status`, whose job shows up as `ci / ci`; (b) a self-contained `ci.yml` running the `chrischall/workflows/.github/actions/arm-gate@main` composite in status mode, whose job shows up as a bare `ci` (mcp-utils, realty-mcp, apple-swift-mcp, curtaincall, …). Both are current and correct — the job name is incidental, only the `ci-gated` status is required. Either way the caller must grant `permissions: statuses: write`, or the gate cannot post at all.

**How the status gate behaves:** un-armed PR → the gate posts **`ci-gated: pending`** (yellow) while the job itself stays green; once `pr-auto-review` passes and arms `ready-to-merge`, that `labeled` event fires the real CI run, which flips `ci-gated` to success. Dependabot PRs run CI unconditionally. **This is strictly stronger than the legacy gate it replaced:** that one left `ci / ci` *skipped* on an un-armed PR, and GitHub treats a skipped required check as **satisfied**, so an admin could click Merge before CI had ever run. A *pending* status is not satisfied, so status mode blocks the button too — the old "convention, not enforcement" caveat is now closed. **But `gate-mode` still defaults to `fail` (legacy) in `reusable-mcp-ci.yml`**, and the `ci-gated` step is conditional on `gate-mode == 'status'`, so a stub that omits the input posts no status at all. Pair the input and the ruleset context, or the repo is bricked.

**Repo setting: Allow auto-merge** — turn it on (`gh api -X PATCH repos/chrischall/<repo> -F allow_auto_merge=true`). This is separate from the workflow and the ruleset, and easy to miss: with it OFF, `auto-merge.yml`'s `gh pr merge --auto` silently fails to arm, so Dependabot / `ready-to-merge` PRs sit OPEN forever even with green CI. The workflow + `RELEASE_PAT` + the `ci-gated` ruleset are all necessary but NOT sufficient without this toggle. (Check the fleet: `gh api repos/chrischall/<repo> --jq .allow_auto_merge`.) **The ruleset makes this toggle load-bearing — applying protection LATER breaks a previously "working" repo:** before the `ci-gated` ruleset exists, armed PRs can merge without the toggle (no required checks pending → the pipeline's merge path succeeds), so the gap hides. The moment `update-ruleset.sh` lands, the next armed PR's arm step fails `GraphQL: Auto merge is not allowed for this repository (enablePullRequestAutoMerge)` and the PR sits green-but-open. Recovery needs NO new PR event (unlike the review step): enable the toggle, then `gh run rerun <failed-auto-merge-run-id> --failed` — the arm step re-executes `gh pr merge --auto` fine on a rerun and merges immediately when checks are already green. Apply ruleset + toggle together; a repo's `update-ruleset.sh` should set `allow_auto_merge=true` alongside the squash-only merge policy so the pair can't drift apart.

**Don't trust a sibling's `scripts/update-ruleset.sh` — check the context it writes before running it.** Only a few repos carry the script, and copies drift from the rulesets they configure: as of 2026-07-19 `alltrails-mcp` and `getyourguide-mcp` both shipped a copy still writing `"context": "ci / ci"` while their own live rulesets were correctly on `ci-gated`. Running such a copy *breaks a working repo* — it replaces the required check with one nothing reports, blocking every PR. Grep the script for `required_status_checks` and confirm it says `ci-gated` before executing it or copying it into a new repo.

**Secrets** — per-repo, set by the **human** (an agent must never set credential values): `CLAUDE_CODE_OAUTH_TOKEN` (review), `RELEASE_PAT` (release-please + the auto-merge/arm step — a `GITHUB_TOKEN`-added label won't fire downstream workflows), optional `CLAWHUB_TOKEN`; plus npm **trusted publishing** for the `--provenance` publish.

**Claude GitHub App — install on the new repo (separate from the secrets).** `pr-auto-review.yml` posts its verdict as `claude[bot]` via the App's OIDC, and the **arm step reads that verdict to add `ready-to-merge`**. If the App isn't installed on the repo, the review job still runs and exits `success` but posts NOTHING and emits no `structured_output` — so the arm step reads `verdict=unknown` and never arms (PR sits green-but-open forever). The `CLAUDE_CODE_OAUTH_TOKEN` only authenticates the model, not repo posting. Human step: github.com/apps/claude → Configure → add the repo. Tell: zero `claude` PR comments = App not installed.

**release-please** — `release-please-config.json` (`package-name`, `release-type: node`, `changelog-sections`, `extra-files`) + `.release-please-manifest.json`. The workflow itself is the thin stub above (`release-please-action` → `mcp-publish` composite); the publish job stays in *this* repo's `release-please.yml` (not a reusable workflow) because npm trusted publishing + `mcp-publisher` bind to this repo's workflow identity via OIDC. **`extra-files` MUST list every file that carries the version**, or `versionSyncTest` fails the release PR's CI: `manifest.json` `$.version`; `server.json` `$.version` + `$.packages[*].version`; `.claude-plugin/plugin.json` `$.version`; `.claude-plugin/marketplace.json` `$.plugins[*].version` + `$.metadata.version`; and **every `src/*.ts` with an `// x-release-please-version` marker**. **Prefer a single `src/version.ts`** (`export const VERSION = '…'; // x-release-please-version`) imported everywhere it's needed — one source, one extra-files entry, nothing to forget — rather than scattering the marker across `index.ts` + a `SERVER_VERSION` in `auth.ts`. Don't hand-bump; release-please does it. **Seed `.release-please-manifest.json` at `0.0.0` on a brand-new package AND set `initial-version`** — seeding a natural-looking `0.1.0` makes the first `feat:` bump to `v0.2.0`, so the first tag skips `v0.1.0` entirely (release-please treats the seed as "already released"). But `0.0.0` alone does **NOT** yield `0.1.0`: with no prior tag, release-please applies its initial-release default and proposes **`1.0.0`** (observed on charlotteonthecheap-mcp 2026-07-19 — the genesis `feat:` produced a `chore(main): release 1.0.0` PR). A brand-new server shipping as 1.0.0 advertises a stable API it has not earned, and the rest of the fleet is on 0.x. So pair the `0.0.0` seed with **`"initial-version": "0.1.0"`** in `release-please-config.json`, and add **`"bump-minor-pre-major": true`** so a later breaking change on 0.x bumps the minor instead of silently declaring 1.0.0. Check the proposed version on the first release PR before merging it — that PR is the last chance to correct it without a bogus tag.

**Publish scaffold** (also list in package.json `files`): `manifest.json` (mcpb; `runtimes.node` floor stays at an **LTS** like `>=22.5`, not 26, so LTS users install), `server.json` (registry; description **≤ 100 chars** or `mcp-publisher` 422s), `.claude-plugin/plugin.json` + `marketplace.json`, `.mcp.json`, `skills/<name>/SKILL.md`, and **`.mcpbignore`** so `mcpb pack` ships only `dist/bundle.js` + `manifest.json` + `package.json` — exclude `src/`, `tests/`, `docs/`, `node_modules/`, `.env*`, the registry/plugin manifests, **and** `release-please-config.json` / `.release-please-manifest.json` / `CHANGELOG.md`.

**mcp-publisher** — install via the shared pinned, SHA-256-verified action `chrischall/workflows/.github/actions/install-mcp-publisher@main`, NOT the upstream `releases/latest | tar xz` (that binary runs with the OIDC + RELEASE_PAT + CLAWHUB tokens).

## Gotchas (hard-won)

- **ESM + NodeNext**: every relative import ends in `.js`, even from `.ts`.
- **`"types": ["node"]`**: NodeNext repos need this in `tsconfig.compilerOptions` once they import a scoped pkg like `@chrischall/mcp-utils`, or `tsc` fails with `Cannot find name 'path'/'url'`. Add `types`, not `typeRoots`.
- **`tsconfig rootDir: "src"`** (not `"."`): with `rootDir:"."` + `include:["src"]`, `tsc` emits `dist/src/index.js` while `package.json` `bin` points at `dist/index.js`, so `npx <svc>` (and any host launching the bin) can't find its entry. Copying a sibling's tsconfig avoids this; a fresh scaffold gets it wrong.
- **The `.mcpb` bundle ships NO `node_modules` — an eager import of an esbuild-`--external` dep crashes it at LOAD.** A top-level `import { X } from 'pkg'` of an externalized/optional dep (e.g. `@fetchproxy/server` in `transport-fetchproxy.ts`) throws `ERR_MODULE_NOT_FOUND` the moment the host spawns the bundled server — before it answers `initialize` (the host logs "Server transport closed unexpectedly", not a useful error). Make optional/externalized deps **lazy**: `import type { X } from 'pkg'` (type-only, erased) + `const { X } = await import('pkg')` inside the method that needs it, so the default path never touches it. `loadDotenvSafely`'s guarded import already does this for `dotenv`; do the same for `@fetchproxy/server` in any cookie-session/bearer repo that keeps fetchproxy as an optional fallback.
- **Add a server-boot smoke test** (`tests/server-boot.test.ts`): spawn the REAL built artifacts — `dist/bundle.js` in a temp dir with NO `node_modules` (the `.mcpb` runtime) and the `bin` `dist/index.js` WITH them — run the `initialize` + `tools/list` handshake, and assert the tools list. Catches both the eager-import crash above and a wrong `bin` path, which unit tests (mocking everything) never see. **Assert `tools.length >= N`, not an exact count** — PR CI runs the branch MERGED with `main`, so a hardcoded `toHaveLength` breaks the instant another PR adds a tool; let `index.test.ts` own the exact roster on its own branch.
- **zod 4**: mcp-utils peer-requires `zod@^4.4.0`. Repos still on zod 3 must bump (schemas are compatible; `z.string().email()` is deprecated-but-works).
- **Never commit secrets**: `.env` must be gitignored. Real creds live in `.env` (local) or the MCP host's `mcp_config.env`. If `.env` is tracked, `git rm --cached` it.
- **`.gitignore` must list `coverage/`** (alongside `dist/`, `node_modules/`, `.env*`, `*.tsbuildinfo`). The minimal `.gitignore` copied across the fleet often omits `coverage/`, so the moment you run `npm run test:coverage` + `git add -A` you silently stage the whole HTML report. The genesis leak-check catches it, but add `coverage/` at scaffold time so it never happens. (`.mcpbignore` already excludes it from the bundle — that's separate from git tracking.)
- **Version sync**: version is duplicated across `package.json`, `src/version.ts` (`x-release-please-version` marker), `manifest.json`, `server.json`, `.claude-plugin/*`. Keep them equal — `versionSyncTest` from `/test` guards it. Don't hand-bump; release-please / the Tag-&-Bump action does it. **`versionSyncTest` flags EVERY line containing the literal `x-release-please-version` — including PROSE.** If a doc comment near `version.ts` mentions the marker string in passing (e.g. "tracked via the `x-release-please-version` marker"), the test reads that line, finds no version literal, and fails with `<no version literal found>`. Keep the marker string off every line except the actual `export const VERSION = '…'; // x-release-please-version` line — reword explanatory comments to not quote it.
- **`server.json` description ≤ 100 chars** (MCP registry schema) — over that, `mcp-publisher publish` 422s.
- **Private repo + npm provenance don't mix.** `npm publish --provenance` (the fleet default — sigstore attestation) only works from a **public** GitHub repo. A private repo 422s at the npm step: *"Unsupported GitHub Actions source repository visibility: 'private'. Only public source repositories are supported when publishing with provenance."* So for a private / personal MCP, pick one: (a) keep it unpublished — `"private": true` in `package.json` + remove the `publish` job from `release-please.yml` (release-please can still own version bumps + tags + the `.mcpb`/GitHub release), (b) drop `--provenance` from the `npm publish` step (you lose the attestation; a public *scope* still publishes fine via `--access public`), or (c) make the repo public. Don't leave the default `--provenance` publish wired on a private repo — every release fails.
- **npm name: check it FIRST, and scope when it's contested.** Good unscoped MCP names are squatted (`gemini-mcp`, `nano-banana-mcp`, `nano-banana-pro-mcp` all taken at 1.x), and even an *available* unscoped name can be **rejected at `npm publish` for being "too similar" to an existing package** (`gemini-image-mcp` collides with `mcp-gemini-image`) — `npm view` shows it free but publish 422s. `npm view <name> version` only proves it's unpublished, NOT publishable. Fix: **publish under the `@chrischall` scope** (`@chrischall/gemini-mcp`) — scoped names are exempt from the similarity check, same as `@chrischall/mcp-utils`. Only the npm-publish identity needs the scope: `package.json` `name` + `publishConfig.access:"public"`, `server.json` `packages[].identifier`, release-please `package-name`, and the install/`npx` commands. Keep `bin`, `mcpName`/`server.json` registry name (`io.github.chrischall/<repo>`), the `.claude-plugin` names, and the repo itself UNSCOPED so the whole identity stays one word. Decide the name (and `gh repo create`) only after `npm view` + a scope decision — renaming a repo + local dir + every manifest afterward is avoidable churn.
- **A `pass` verdict's prose nits are tracked NOWHERE — only `warn`/`fail` open the follow-up issue.** The auto-review pipeline files/updates the `Auto-review follow-ups for PR #N` issue only when the verdict is `warn` or `fail`; a `pass` whose verdict text still mentions a nit ("one nit: …") leaves it untracked, and the PR auto-merges seconds later. When a pass verdict carries prose nits, file the follow-up issue (or just fix it in a follow-up PR) yourself — nothing in the pipeline will remember it for you.
- **Re-running `pr-auto-review` does NOT re-arm; trigger a fresh PR event.** `gh run rerun` re-executes the review job but `claude-code-action` does not regenerate `structured_output` on a rerun (the "Surface verdict" step shows `skipped`), so the arm step falls back to `verdict=unknown` and won't add `ready-to-merge`. To get a clean verdict + arm, fire a fresh `pull_request` event — `gh pr close <n> && gh pr reopen <n>` (or a push) — not a workflow rerun.
- **A PR that CHANGES a workflow file can't be auto-reviewed — the Claude App's OIDC token exchange validates the workflow against `main`.** The App's app-token exchange fails `401 — "Workflow validation failed. The workflow file must exist and have identical content to the version on the repository's default branch"` whenever the PR's `pr-auto-review.yml` (or any workflow it touches) differs from the copy on the default branch. The review step then emits NO verdict (the log even says *"this is normal … you should ignore this error"*), so the PR shows `review/review → fail` / "no verdict" and never arms. This is a **chicken-and-egg every repo hits when first adopting the shared workflows** (the PR that swaps `pr-auto-review.yml` to the stub can't be reviewed by the new-but-not-yet-on-`main` workflow), and it recurs for any later workflow-only PR. Tell it apart from a real auth/token failure: the review fast-fails in ~20–30 s with the `App token exchange failed: 401 … Workflow validation failed` line, while other repos' auto-reviews still pass (so the token is fine). **Resolution: this PR must be merged ONCE outside the auto-review path** — have the human add `ready-to-merge`, which fires the deferred CI run and flips `ci-gated` to success, then auto-merge takes it (no verdict needed). Under status-mode gating the label is the ONLY route: an un-armed PR sits on `ci-gated: pending`, and a *pending* required status blocks the merge button, so the old "just squash-merge it manually" escape (which relied on a *skipped* check counting as satisfied) no longer works. After it lands on `main`, every subsequent PR auto-reviews normally. Per the never-merge rule, surface this and let the human arm/merge — don't add `ready-to-merge` yourself.
- **stdio transport**: logs go to **stderr only** — stdout is reserved for JSON-RPC.
- **Don't merge PRs**: open with one release-notes label; `pr-auto-review` + `auto-merge` ship it. Never add `ready-to-merge` yourself to override a warn/fail.
- **Auto-merge ships a PR out from under you — one complete commit, no follow-ups.** `pr-auto-review` + `auto-merge` squash-merge a PR the moment its review flips to `pass` (often < 1 min after opening). Any commit you push to the branch AFTER that lands on a now-stale branch and **orphans — it never reaches `main`** (this keeps biting: a "one more line" follow-up push silently vanishes). So open a PR only when the change is COMPLETE in a single push; never use a PR branch as a staging area for follow-ups. Left something out? Verify it actually landed (`git branch -r --contains <sha> | grep origin/main`), then it's a NEW branch + PR — not another push to the merged one. Need a real checkpoint without shipping? Open it `--draft` (auto-review skips drafts).
- **A prior PR squash-merging out from under you leaves your follow-up branch with DUPLICATE commits — replay onto fresh `main`, don't PR the stale branch.** If you keep committing locally on `feat/X` while PR #N (off the same base) auto-merges, your branch's merge-base is now behind `main`, and `main` holds ONE squash commit while your branch still has the original individual commits of that merged work. Opening a PR from it (or pushing) re-creates a deleted branch and offers a diff full of already-merged churn. Recovery: `git fetch`; branch off current `origin/main`; replay ONLY the genuinely-new commits — `git rebase --onto origin/main <last-already-merged-sha>` (or `git checkout -b feat/X2 origin/main && git cherry-pick <last-merged-sha>..HEAD`); the cherry-picks apply cleanly because the squash's tree == the last-merged commit's tree. Then `git push origin --delete <stale-branch>` and open the PR from the clean one. (Confirms: `git log --oneline origin/main..HEAD` should show ONLY your new commits.)
- **Parallel feature PRs collide on `index.ts` + `index.test.ts`** — the tool-registrar list and the tool-count/sorted-roster are serial conflict magnets. Branch each feature off `main` (don't stack PRs — a squash-merge gives the parent a new SHA and the stacked branch then carries a duplicate of it), and expect to **rebase the second PR** after the first merges: `git reset --hard origin/main` + cherry-pick the feature commit (or `git rebase origin/main`), union-merge the two registrars, bump the count. Trivial, but it WILL conflict — don't be surprised.

## New MCP — fast path

0. **First: is a full MCP even wanted?** Default to shipping the fpx skill only (see *Deliverable: fpx skill first* above) — ASK if it wasn't dictated, CONFIRM if it was, and ship the fpx skill either way. Only continue this fast path once a full MCP is agreed.
1. **Lock the name before creating anything.** `npm view <name> version` for each candidate (and watch the similarity trap — see Gotchas); if contested, publish under `@chrischall/<name>`. Settle repo name + npm name together, THEN `gh repo create` — renaming later cascades across the repo, local dir, and every manifest.
2. Copy a same-archetype sibling's skeleton (splitwise for bearer, redfin for fetchproxy) — packaging, tsconfig, vitest from a local clone is fine, but **pull `.github/workflows/*.yml` from a freshly-updated sibling's `origin/main`, not the local tree** (see Workflows above — a stale local clone hands you the old self-contained workflows). Then do the full **Repo bootstrap** (workflows, labels incl. `release-ready`, rulesets with the `ci-gated` status check, secrets + **Claude GitHub App install**, release-please extra-files, publish scaffold, `.mcpbignore`) — it's the part that's easy to half-do. Expect the genesis push to `main` to be direct (it puts the workflows on the default branch so PRs can run); creating the **public** repo triggers an explicit auth confirmation from the user — ask before `gh repo create --public`.
3. Add `"@chrischall/mcp-utils"` (published `^x` once available; `file:../mcp-utils/<tarball>` pre-publish).
4. Write `client.ts` (deferred-config-error + one central `write()`), `tools/*.ts` (`registerTool` + `textResult`, **`confirm`-gated** writes with a dry-run `preview()`), wire `index.ts` with `runMcp`. TDD; verify endpoints from a capture before coding writes.
5. Tests with `createTestHarness` + `versionSyncTest`; mock the network. Keep them green.
6. `npm run build && npm test`. Branch + PR; let auto-merge ship it (never merge it yourself). Ship the **fpx skill** alongside (it's the minimum deliverable — never skipped, even for a full MCP).
